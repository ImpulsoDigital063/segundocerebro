## O que foi feito — AgendaPRO Security Hardening

### Auditoria de segurança (3 agentes em paralelo)
Rodei uma auditoria com 3 frentes simultâneas:
1. **API routes** — autenticação, IDOR, injeção, rate limiting
2. **Client-side** — XSS, secrets expostos, CSRF, validação de input
3. **Auth & data flow** — middleware, tokens, ownership checks, price tampering

### Vulnerabilidades críticas encontradas e corrigidas

| Vulnerabilidade | Risco real | Fix |
|---|---|---|
| Token HMAC com fallback `'fallback-secret'` | Qualquer um forja tokens | Exige env var obrigatória, hash completo |
| Timing attack na verificação de token | Hacker descobre token char por char | `timingSafeEqual()` |
| XSS em templates de email | Phishing via nome de cliente | `escapeHtml()` em todo input |
| `/api/claim-review` sem auth | Fraude infinita de pontos | Auth + ownership check |
| Senha temporária exposta na API response | Interceptação = account takeover | Senha vai por email, nunca na response |
| IDOR em `/api/notify` | User A acessa dados do User B | Ownership check `owner_id === user.id` |
| Cron routes com anon key | Lembretes nunca seriam enviados | Service role client |
| Sem rate limiting | Brute force em cadastro, senha, pontos | Rate limiter in-memory por IP |
| Sem CSP header | Script injection | CSP + frame-ancestors none |
| Double-booking possível | 2 clientes no mesmo horário | Trigger SQL + check client-side |

### Migrations rodadas no Supabase
- **V8** — RLS restritivo pra `clients` e `waitlist`
- **V9** — Trigger `check_appointment_overlap` que barra INSERT/UPDATE com horário conflitante

## O que aprendi — Padrão de Auditoria de Segurança

**Insight principal:** Auditar não é "ler o código e achar bugs". É pensar como atacante: "se eu quisesse roubar dados, manipular preços, ou derrubar o sistema, por onde eu entraria?"

### Checklist que funcionou (salvo como padrão em `3-RETENCAO/padroes/`)
1. **API routes** — toda rota POST/PATCH precisa de auth. Toda rota que acessa dados precisa de ownership check.
2. **Tokens** — nunca truncar hash, nunca usar fallback previsível, sempre `timingSafeEqual`.
3. **Email/HTML** — todo input de usuário que vira HTML precisa de escape. Sem exceção.
4. **Rate limiting** — endpoints públicos e de autenticação são alvos de brute force.
5. **Supabase** — anon key + RLS = funciona pra client. Crons e admin operations = service role.
6. **select('*')** — em páginas públicas expõe campos internos. Listar só o necessário.
7. **Senhas** — mínimo 8 chars + complexidade. Temporárias geradas com crypto, nunca Math.random.
8. **Error messages** — nunca vazar mensagem do banco pro client. Log no server, mensagem genérica pro front.

### O que não sabia antes de hoje
- `timingSafeEqual` existe no Node.js nativo — comparação de strings em tempo constante
- CSP com Next.js precisa de `'unsafe-inline'` pra styles (Next usa inline styles em todo lugar)
- Vercel Cron não manda cookies — route que usa `createClient()` (anon) não funciona pra cron
- Trigger no PostgreSQL é a forma definitiva de evitar race conditions (overbooking)

## O que aprendi

- **Domínios .br levam até 24-48h** pra propagar nameservers — diferente de .com que é quase instantâneo
- **Vercel Hobby** limita crons a 1x/dia — precisa do Pro pra rodar a cada 15min
- **`new Resend()` no top-level** quebra o build porque Next.js importa o módulo durante o build e a env var não existe
- **Mercado Pago usa `preapproval`** pra assinaturas recorrentes, não `subscription`
- **Feedback de design** da namorada: se o usuário não entende como usar, o produto não tá pronto. UX > features.
